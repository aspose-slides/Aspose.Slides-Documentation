- 보안
- 안정성
- 확장성/속도
- 가격
- 기능

## **보안**
다음은 앞서 언급된 Microsoft 기사에서 직접 인용한 내용입니다: 
*"Office 응용 프로그램은 서버 측에서 사용하도록 설계되지 않았으며, 따라서 분산 구성 요소가 직면하는 보안 문제를 고려하지 않습니다. Office는 들어오는 요청을 인증하지 않으며, 서버 측 코드에서 실수로 매크로를 실행하거나 매크로를 실행할 수 있는 다른 서버를 시작하는 것으로부터 보호하지 않습니다. 익명 웹에서 서버로 업로드된 파일을 열지 마십시오! 마지막으로 설정된 보안 설정에 따라 서버는 관리자 또는 시스템 컨텍스트에서 전체 권한으로 매크로를 실행하여 네트워크를 위험에 빠뜨릴 수 있습니다! 또한 Office는 처리 속도를 높이기 위해 클라이언트 인증 정보를 캐시할 수 있는 Simple MAPI, WinInet, MSDAIPP와 같은 다수의 클라이언트 측 구성 요소를 사용합니다. Office가 서버 측에서 자동화되는 경우, 하나의 인스턴스가 여러 클라이언트를 서비스할 수 있으며, 해당 세션에 대해 인증 정보가 캐시되었기 때문에 한 클라이언트가 다른 클라이언트의 캐시된 자격 증명을 사용하여 다른 사용자를 가장함으로써 부여되지 않은 접근 권한을 얻을 수 있습니다."*

Aspose 제품은 매우 안전합니다. 따라서 Aspose 구성 요소는 중요한 시스템 리소스에 잠재적인 위험을 초래하지 않습니다. 또한 Aspose 구성 요소가 문서를 열 때 매크로가 자동으로 실행되지 않습니다. Aspose 구성 요소는 개발자가 Office 파일을 생성, 조작 및 저장할 수 있도록 설계되었습니다. Microsoft Office 패키지와 관련된 위험은 Aspose 구성 요소에 내재되어 있지 않습니다.

## **안정성**
다음은 앞서 언급된 Microsoft 기사에서 직접 인용한 내용입니다: 
*"Office 2000, Office XP 및 Office 2003은 Microsoft Windows Installer(MSI) 기술을 사용하여 최종 사용자가 설치 및 자동 복구를 더 쉽게 할 수 있도록 합니다. MSI는 '첫 사용 시 설치' 개념을 도입하여 기능을 런타임에 동적으로 설치하거나 구성할 수 있게 합니다(시스템 전체 또는 주로 특정 사용자에 대해). 서버 측 환경에서는 이는 성능을 저하시킬 뿐만 아니라 사용자가 설치를 승인하거나 적절한 설치 디스크를 제공하도록 요청하는 대화 상자가 나타날 가능성을 높입니다. Office를 최종 사용자 제품으로서 회복력을 높이기 위해 설계되었지만, Office의 MSI 기능 구현은 서버 측 환경에서는 역효과를 냅니다. 또한 Office 전반의 안정성은 서버 측에서 실행될 때 보장될 수 없으며, 이는 해당 용도로 설계·테스트되지 않았기 때문입니다. 네트워크 서버에서 Office를 서비스 구성 요소로 사용하면 해당 머신의 안정성이 떨어지고 결과적으로 전체 네트워크의 안정성도 영향을 받습니다. 서버 측에서 Office를 자동화하려면, 중요한 기능에 영향을 주지 못하고 필요할 때 재시작할 수 있는 전용 컴퓨터로 프로그램을 격리하십시오."*

Aspose 구성 요소는 단일 DLL에 패키징되어 있기 때문에 기능을 위해 추가적인 부분이나 조각을 설치할 필요가 전혀 없습니다. Aspose 구성 요소는 C++ 애플리케이션에서만 사용되며, 인간의 응답을 기다리도록 설계된 코드 부분이 없습니다. Aspose 구성 요소는 철저히 테스트되었으며 매우 안정적입니다. Aspose 구성 요소는 [Companies](https://about.aspose.com/customers)와 같은 **IBM**, **Hilton**, **Reader's Digest**, **Bank of America** 등 많은 기업에서 사용됩니다.

## **확장성/속도**
다음은 앞서 언급된 Microsoft 기사에서 직접 인용한 내용입니다: 
*"서버 측 구성 요소는 다중 클라이언트를 위해 최소 오버헤드와 높은 처리량을 가진 고도로 재진입 가능하고 다중 스레드 COM 구성 요소여야 합니다. Office 응용 프로그램은 거의 모든 면에서 정반대입니다. 이들은 재진입이 불가능하고 STA 기반 자동화 서버이며, 단일 클라이언트를 위한 다양하지만 자원 집약적인 기능을 제공하도록 설계되었습니다. 서버 측 솔루션으로서 확장성이 거의 없으며 메모리와 같은 중요한 요소에 고정된 제한이 있어 구성으로 변경할 수 없습니다. 더욱이 이들은 전역 리소스(예: 메모리 매핑 파일, 전역 추가 기능 또는 템플릿, 공유 자동화 서버)를 사용하므로 동시에 실행될 수 있는 인스턴스 수가 제한되고 다중 클라이언트 환경에서 구성될 경우 경쟁 조건이 발생할 수 있습니다. 여러 Office 응용 프로그램 인스턴스를 동시에 실행하려는 개발자는 잠재적인 교착 상태나 데이터 손상을 방지하기 위해 Office 응용 프로그램에 대한 풀링 또는 직렬화 접근을 고려해야 합니다."*

Aspose 구성 요소는 매우 확장 가능하고 번개처럼 빠릅니다. Office 응용 프로그램은 수백, 수천 명의 사용자가 동시에 사용하도록 설계되지 않았습니다. 그러나 Aspose 구성 요소는 바로 이를 위해 설계되었습니다. 우리의 구성 요소는 진정한 C++ 솔루션이며 단일 서버에서 단일 애플리케이션을 구동하든, 로드 밸런싱된 웹 폼에서 전사 수준 애플리케이션을 구동하든 완벽히 작동합니다.

## **가격**
애플리케이션이 Microsoft Office 자동화를 사용할 경우, 해당 애플리케이션이 실행되는 각 머신마다 Microsoft Office 사본을 구매해야 합니다. 종종 애플리케이션이 Office 파일을 생성하거나 조작해야 하지만 사용자가 Microsoft Office를 보유할 필요가 없는 경우가 있습니다. Aspose는 [Cost Effective](https://purchase.aspose.com/)하고 로열티가 없는 재배포 라이선스를 제공하여 라이선스 걱정 없이 무제한 사용자에게 배포할 수 있게 합니다. 웹 기반 애플리케이션을 만들 때 Microsoft Office 자동화 구성 요소는 서버 측 솔루션용으로 가격이 매겨지거나 라이선스가 제공되지 않으므로, Microsoft Office 구성 요소를 활용하는 웹 애플리케이션을 배포할 적절한 라이선스 솔루션이 없습니다. Aspose는 서버 기반 애플리케이션을 위한 매우 [Cost Effective](https://purchase.aspose.com/) 솔루션도 제공합니다.

## **기능**
Aspose 구성 요소는 Office 파일 관리에 필요한 모든 것을 제공하며 그 이상을 제공합니다. 최소한의 작업으로 최고의 결과를 얻을 수 있도록 설계되었습니다. Office 자동화와 달리 Aspose 구성 요소는 강력하고 시간 절약 기능을 많이 제공합니다. 예를 들어, [Aspose.Cells](https://products.aspose.com/cells/cpp/)은 개발자가 **DataTable** 또는 **DataView**의 데이터를 직접 Excel 파일로 가져올 수 있게 합니다. [Aspose.Words](https://products.aspose.com/words/net/)는 개발자가 C++ 데이터 객체에서 직접 Word(메일 머지) 문서를 채울 수 있는 유사한 기능을 제공합니다. Aspose 패밀리의 [Every Component](https://products.aspose.com/total/cpp/)은 각각 고유하고 강력한 기능 세트를 제공합니다. Aspose 구성 요소를 구매하면 개발 팀에 접근할 수 있다는 것이 가장 큰 장점입니다. 우리 개발 팀은 귀사의 필요 기능이 다른 회사에도 필요할 가능성이 높다는 것을 인식하고 있습니다. 모든 기능 요청을 추가할 수는 없지만, 팀은 지원을 제공할 때 매우 열린 마음과 유연성을 가지려고 노력합니다. 이러한 사고방식이 Aspose 구성 요소를 지금처럼 강력하게 만든 이유입니다. Office 자동화 객체에서 추가 기능이 필요한다면, 해당 기능이 추가될 가능성은 매우 낮습니다.
